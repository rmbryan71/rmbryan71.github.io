# Claude Guidelines — _outlines/

Outlines translate an idea into a structure. Every section should earn its place — if it doesn't serve the non-technical reader's understanding, cut it.

## When Helping with an Outline
- Each section heading should map to one clear idea or step
- For every technical term that will appear in the draft, note its plain-language definition here so it's consistent throughout
- Estimate word count per section to stay under the 750-word total (rough guide: intro ~75, each body section ~150, closing ~75)
- Explicitly mark where visuals should go and describe what they should show — don't leave this for the draft stage

## Outline File Format
```
# [Title]

**Target audience:** Non-technical readers
**Estimated word count:** [X] / 750

## Terms to Define
- [term]: [plain-language definition]

## Structure

### Intro (~75 words)
Hook + what the reader will understand by the end

### [Section 1 Title] (~XXX words)
- Key point
- Key point
- <!-- VISUAL OPPORTUNITY: [what the diagram/chart shows] -->

### [Section 2 Title] (~XXX words)
- Key point
- Key point

### Closing (~75 words)
Takeaway — what should stick with the reader?

## Open Questions / Research Needed
```

## What to Flag
- Sections that assume background knowledge — note what needs to be explained first
- Any section that feels like it could be its own article
- Places where the logical flow may lose a non-technical reader
- Visual opportunities that could replace an entire prose section
